# Problem 112
## Domain
```
The Blocksworld domain consists of a table and a set of blocks. Each block can be placed either directly on the table or on top of another block. The robotic arm can hold only one block at a time. The actions available in this domain include picking up a block from the table, placing a block on the table, stacking one block on top of another, and unstacking a block from another block. Each action has specific preconditions that must be met before it can be performed, such as ensuring that a block is clear (no block on top of it) or that the robotic arm is not currently holding any block. The effects of these actions change the state of the blocks and the arm's holding status accordingly.
```

## Problem
```
The initial state of the world includes five blocks: black_block_1, brown_block_1, brown_block_2, orange_block_1, and black_block_2. The initial configuration has black_block_1, brown_block_1, and orange_block_1 on the table, while brown_block_2 is on top of black_block_1 and black_block_2 is on top of orange_block_1. The goal is to make orange_block_1 clear, meaning there should be no block on top of it. The robotic arm starts not holding any block.
```

## Actions
```
1. **pickup x**: This action allows the robotic arm to pick up block x from the table. You can only do this if block x is clear (no block on top of it), it is on the table, and you are not currently holding any block. After picking up, you will be holding block x and it will no longer be on the table.

2. **putdown x**: This action lets you place block x on the table. You can only perform this if you are currently holding block x. After putting down, block x will be on the table, clear, and you will no longer be holding it.

3. **stack x y**: This action allows you to place block x on top of block y. You can only do this if you are holding block x and block y is clear (no block on top of it). After stacking, block x will be on top of block y, block x will be clear, block y will no longer be clear, and you will not be holding block x anymore.

4. **unstack x y**: This action lets you pick up block x from the top of block y. You can only perform this if block x is on top of block y, block x is clear, and you are not holding any block. After unstacking, block x will no longer be on top of block y, block y will be clear, and you will be holding block x.
```

## Goal
```
Make orange_block_1 clear, meaning there should be no block on top of it.
```

## Constraints
```
['There must be at least one state where there is a block on top of black_block_2, meaning black_block_2 is not clear.', 'If there is a block on top of black_block_2, then there must have been an earlier state where either brown_block_1 is on top of brown_block_2 or black_block_1 is clear.']
```

```

```

## Reasoning
```
Constraint: There must be at least one state where there is a block on top of black_block_2, meaning black_block_2 is not clear.
The goal is to make orange_block_1 clear, which means that there should be no block on top of it. This can be represented as 'clear(orange_block_1)'. However, the goal is not included in the final LTL formula as per the instructions. 

The constraint states that there must be at least one state where there is a block on top of black_block_2, meaning black_block_2 is not clear. This is a PRECEDENCE constraint because it requires that at some point in the future, the condition of having a block on top of black_block_2 must hold true. This can be expressed as 'F(!clear(black_block_2))', which means that eventually, black_block_2 will not be clear (i.e., there will be a block on top of it). 

Combining these, the LTL formula will include the constraint that black_block_2 is not clear at some point in the future, while the goal of making orange_block_1 clear is stated separately.

Constraint: If there is a block on top of black_block_2, then there must have been an earlier state where either brown_block_1 is on top of brown_block_2 or black_block_1 is clear.
The constraint provided is a consequence constraint because it specifies a condition that must hold true based on the state of the blocks. Specifically, it states that if there is a block on top of black_block_2, then there must have been a previous state where either brown_block_1 is on top of brown_block_2 or black_block_1 is clear. 

To express this in LTL, we need to capture the following: 
1. The condition of having a block on top of black_block_2, which can be expressed as `onTop(x, black_block_2)` for some block x. 
2. The requirement that in some earlier state (denoted by the use of the `F` operator), either `onTop(brown_block_1, brown_block_2)` or `clear(black_block_1)` must hold. 

Thus, we can express this as: 
- G(onTop(x, black_block_2) -> F(onTop(brown_block_1, brown_block_2) | clear(black_block_1)) ). This means that globally, if there is a block on top of black_block_2, then eventually (in some future state), either brown_block_1 is on top of brown_block_2 or black_block_1 is clear. 

This captures the necessary condition for the constraint while ensuring that the logical structure of the LTL formula is maintained.
```

## LTL Formula
```
(F(!clear(black_block_2))) AND (G(onTop(x, black_block_2) -> F(onTop(brown_block_1, brown_block_2) | clear(black_block_1))))
```
