# LAPIS: Code Pipeline Architecture and Methodology

This document thoroughly explains the internal workings of the LAPIS (Language-Adaptive PDDL Iterative Synthesis) code pipeline as used in the experimental benchmarks. It is detailed for the purposes of code reproducibility, theoretical comparison with other works (e.g., standard LLM+P paradigms), and explanation in academic papers.

## 1. High-Level Overview

LAPIS adopts an **Iterative Synthesis Architecture** (`LowLevelPlanningPipeline`) that bridges the gap between unstructured Natural Language (NL) instructions and classical planning execution through self-reflective PDDL synthesis. 

Critically, LAPIS operates in a **Self-Consistent Synthesized Sandbox**. The LLM generates both the domain and the problem without observing the human-authored Ground Truth (GT) domain (preventing prompt leakage), and relies on external formal verifiers (VAL) to refine its own internal world model.

---

## 2. The Iterative PDDL Refinement Loop

The core of the LAPIS benchmark execution relies on translating natural language objectives directly into actionable, formally verified PDDL structures.

### 2.1 Domain and Problem Generation
- **Domain Generation**: The LLM synthesizes a `generated_domain.pddl` representing the physics and state of the environment.
- **Problem Generation**: The LLM then generates a compatible `problem.pddl` detailing the specific objects, initial states, and goal objectives.
- **Condition Full Synthesis**: In our most rigorous evaluations, no human-written PDDL schema is provided to the agent in this phase (preventing "Oracle Leakage").

### 2.2 The "Adequacy" Chain-of-Thought
- **Adequacy Check (`check_domain_adequacy` & `check_problem_adequacy`)**: A crucial differentiator for LAPIS. Before sending the generated files to the solver, an explicit LLM reflection step is invoked to check if the generated `domain.pddl` has sufficient predicates and actions to express the raw natural language observation. It allows for on-the-fly amendments to the syntax structure before compute is wasted on a doomed classical planning attempt.

### 2.3 Execution & Formal Validation
The pipeline makes a call to a classical planner (e.g., Fast Downward / `up_fd`):
1. **Plan Validation (`VAL_validate`)**: The output plan is strictly validated using VAL Validator to ensure syntax conformity.
2. **Grounding Validation (`VAL_ground`)**: The plan is verified for correct object/fluent grounding in reality.

### 2.4 The Refinement Loop
If planning fails (e.g., due to syntax errors in the PDDL, unsolvable states, or grounding failures):
- The detailed error trace (Fast Downward logs, pddlenv error logs, and VAL validation/grounding logs) is fed back to the LLM agent.
- The agent iteratively refines (`refine_problem`) the PDDL structures and re-runs the planner up to a maximum number of `pddl_gen_iterations`.
- **Significance for Paper Comparisons**: Standard LLM+P approaches are single-shot and fail catastrophically when they misalign a single predicate name between the domain and problem in synthesis tasks. The self-guided VAL refinement loop is what enables LAPIS to autonomously recover and achieve high success rates in "Full Synthesis" setups (bridging the "Fairness Gap").

---

## 3. Evaluation Metrics and Verification

Evaluating plans across the IPC benchmark at scale requires rigorous outcome tracking.

- **Outcome Logging**: Throughout the iterative process, performance metrics (such as the number of refinements, plan lengths, domain/problem generation times, failure stages, and failure reasons) are recorded per problem instance into standardized CSV formats for empirical visualization.
- **Ground Truth Oracle**: At the very end of the line, the final valid plan generated by the synthesized world is verified against the compiled, human-authored `compiled_domain.pddl` and `compiled_problem.pddl`. The Ground Truth is strictly walled off during generation and used *only* as an Oracle scoring mechanism for final Table 1 success metrics, protecting the integrity of the evaluation.

---

## 4. Summary for System Comparisons

When comparing LAPIS against baseline LLM+P, SayCan, or ReAct models in literature:
1. **It is an Iterative Synthesizer:** Unlike Zero-Shot LLM+P baselines, LAPIS generates its own PDDL Domain alongside the Problem and iteratively checks its consistency with VAL tools *prior* to returning the final output, overcoming the classical *Domain-Problem Coupling Problem*.
2. **Adequacy Pre-computation:** It is not purely "generate and pray." The system structurally ensures text-to-code alignment via the explicit Adequacy Verification check step before initiating compute-expensive classical planning loops.
3. **Resilience to Model Hallucinations:** By outsourcing functional validation to a formal syntax checker (VAL), the LLM's capacity is utilized entirely for reasoning and structural repair instead of relying entirely on its zero-shot coding accuracy.
